# SaaS-план для Ozon FBO Supply Automation

Цель: превратить локальный MVP в полноценный платный веб-сервис, где пользователь регистрируется, подключает Ozon API, загружает Excel, получает файлы поставки, создаёт черновики FBO в Ozon и может запускать автоматический поиск/бронирование слотов поставки.

## Продукт

### Для кого

- продавцы Ozon FBO;
- менеджеры маркетплейсов;
- агентства, ведущие несколько кабинетов Ozon.

### Главная ценность

Сервис экономит время на подготовке поставок: сам находит товары, распределяет количество по городским кластерам, создаёт XLSX по шаблону Ozon, может создавать черновики поставок и отдельной задачей ловит свободные слоты по нужным складам.

### MVP для продажи

1. Регистрация и вход.
2. Личный кабинет.
3. Подключение одного или нескольких Ozon-кабинетов.
4. Проверка API-ключей.
5. Загрузка Excel.
6. Расчёт распределения.
7. Скачивание ZIP.
8. Создание черновиков Ozon после подтверждения пользователя.
9. Модуль `Слоты`: фоновые задачи поиска и бронирования слотов по складам.
10. История обработок и история попыток бронирования.
11. Тарифы и лимиты.
12. Подписка и блокировка лимитов при неоплате.

## Тарифы

Стартовая сетка для проверки спроса:

| Тариф | Цена | Лимит | Для кого |
| --- | ---: | --- | --- |
| Start | 2 990 ₽/мес | 20 обработок или 5 000 SKU-строк/мес | небольшой продавец |
| Pro | 7 990 ₽/мес | 200 обработок или 50 000 SKU-строк/мес | активный продавец |
| Agency | 19 990 ₽/мес | 1 000 обработок или 250 000 SKU-строк/мес | агентства |

Отдельно можно продавать разовые пакеты обработок, если пользователи не хотят подписку.

Автобронирование слотов лучше оставить для тарифов `Pro` и `Agency`, потому что это самая ценная и самая чувствительная к лимитам Ozon функция. На `Start` можно дать режим "найти слот и уведомить".

## Архитектура

### Frontend

- Next.js 14.
- Страницы:
  - `/` лендинг;
  - `/login`;
  - `/register`;
  - `/app` рабочий экран поставок;
  - `/app/ozon-accounts`;
  - `/app/history`;
  - `/app/slots`;
  - `/app/billing`;
  - `/app/settings`.

### Backend

- FastAPI.
- PostgreSQL для пользователей, организаций, подписок, Ozon-кабинетов и истории задач.
- Redis для очередей и rate limit.
- Celery/RQ/Arq для фоновой обработки больших Excel, API Ozon и задач поиска слотов.
- S3-совместимое хранилище для входных/выходных файлов: Yandex Object Storage, Selectel, S3.

### Модуль бронирования слотов

- Запускается только после явного подтверждения пользователя.
- Создаёт отдельную цель на каждый город/склад из распределения.
- Проверяет цели параллельно, но через общий rate limit на Ozon-кабинет.
- При `429` делает backoff и не ломает всю задачу.
- Для новых товаров сначала расширяет географию через крупные склады: Москва, Санкт-Петербург, Казань, Ростов, Екатеринбург, Воронеж.
- Для товаров с продажами использует потребность по складам и пропускает направления без потребности.
- Поддерживает режимы `только уведомить` и `автоматически бронировать`.
- Подробная спецификация: [SLOT_BOOKING_AUTOMATION.md](SLOT_BOOKING_AUTOMATION.md).

### Безопасность

- Пароли хранить только как hash: Argon2 или bcrypt.
- Ozon Api-Key хранить только зашифрованным: AES-GCM/Fernet/KMS.
- Никогда не писать Api-Key в логи.
- Каждое действие привязано к `organization_id`.
- Для создания черновиков требовать явное подтверждение в UI.
- Для автопоиска слотов требовать отдельное подтверждение, настройку лимитов и кнопку остановки.
- Для публичного сервиса включить HTTPS, CORS только на свой домен, rate limit и audit log.

## База данных

Основные таблицы:

- `users` — пользователи.
- `organizations` — компании/рабочие пространства.
- `organization_members` — роли пользователей в компании.
- `ozon_accounts` — подключённые кабинеты Ozon.
- `subscription_plans` — тарифы.
- `subscriptions` — текущая подписка компании.
- `usage_counters` — месячные лимиты.
- `processing_jobs` — история загрузок Excel.
- `processing_job_files` — созданные XLSX/ZIP.
- `ozon_drafts` — созданные черновики Ozon.
- `slot_booking_jobs` — фоновые задачи поиска/бронирования слотов.
- `slot_booking_targets` — цели задачи по городам/складам.
- `slot_booking_target_items` — SKU и количество внутри цели бронирования.
- `slot_booking_attempts` — журнал запросов и попыток по слотам.
- `audit_events` — журнал важных действий.

## Подписки и платежи

Для российского рынка в первую очередь стоит проектировать интеграцию с ЮKassa или CloudPayments. Для зарубежных клиентов можно держать Stripe как отдельный провайдер.

Абстракция в коде:

```text
BillingProvider
  create_checkout_session(...)
  handle_webhook(...)
  cancel_subscription(...)
  get_customer_portal_url(...)
```

Сначала реализовать один провайдер. Не смешивать бизнес-логику тарифа с конкретной платёжной системой.

## Лимиты

Лимитировать лучше не только пользователей, а именно организацию:

- количество обработок в месяц;
- количество строк SKU в месяц;
- количество подключённых Ozon-кабинетов;
- создание черновиков Ozon;
- количество задач автопоиска слотов;
- хранение истории, например 30/90/365 дней.

При каждой обработке:

1. Проверить подписку.
2. Проверить лимит строк.
3. Создать `processing_job`.
4. Списать usage.
5. Обработать Excel.
6. Сохранить результат и ссылку на скачивание.

## Этапы разработки

### Этап 1. SaaS-фундамент

- PostgreSQL.
- Регистрация, вход, JWT/cookie-сессия.
- Организация по умолчанию для нового пользователя.
- Сохранение Ozon-ключей в базе в зашифрованном виде.
- Перенос текущей обработки Excel в endpoint, требующий авторизацию.

### Этап 2. Рабочий кабинет

- Экран подключения Ozon.
- Экран загрузки Excel.
- История обработок.
- Скачивание старых ZIP.
- Создание черновиков Ozon из результата.
- Экран `Слоты`: запуск поиска, статусы по складам, журнал попыток, остановка задачи.

### Этап 2.5. Автоматизация слотов

- Перевести создание черновиков на актуальный v2/v3-контур Ozon Supply API.
- Добавить `slot_booking` endpoints.
- Добавить очередь Redis и worker.
- Реализовать rate limiter на `ozon_account_id`.
- Реализовать выбор слота по правилам склада, даты и количества SKU.
- Добавить уведомления об успешном бронировании.

### Этап 3. Деньги

- Тарифы.
- Checkout.
- Webhook платежного провайдера.
- Автоматическое продление.
- Блокировка сверх лимита.
- Страница billing.

### Этап 4. Продакшен

- Домен.
- HTTPS.
- Объектное хранилище.
- Очереди.
- Логи и мониторинг.
- Резервные копии PostgreSQL.
- Политика конфиденциальности, оферта, согласие на обработку персональных данных.

## Что нельзя делать

- Нельзя хранить Ozon Api-Key открытым текстом.
- Нельзя создавать поставки/черновики без подтверждения пользователя.
- Нельзя запускать бесконечный агрессивный поиск слотов без rate limit, backoff и ограничения по времени.
- Нельзя бронировать слот в автоматическом режиме без отдельного согласия пользователя.
- Нельзя запускать публичный сервис на portable `server.js`.
- Нельзя делать оплату без webhook-проверки на backend.
- Нельзя доверять тарифу, пришедшему с frontend.

## Ближайший технический шаг

Перевести проект с текущего локального режима в SaaS-режим:

1. Добавить PostgreSQL в `docker-compose.yml`.
2. Добавить SQLAlchemy/Alembic в backend.
3. Реализовать `auth` endpoints.
4. Реализовать `ozon_accounts` endpoints.
5. Сделать frontend-страницы регистрации, входа и личного кабинета.
6. После этого подключать платежи.
