# Автоматизация бронирования FBO-слотов

Цель модуля: после загрузки Excel и расчёта распределения сервис создаёт задачи на поиск слотов поставки и параллельно проверяет доступность по выбранным городам/складам, пока не найдёт подходящий слот или пока задача не будет остановлена.

## Что именно делает модуль

1. Берёт уже распознанные SKU Ozon и количества из Excel.
2. Использует рассчитанное распределение по городским кластерам: Москва, Санкт-Петербург, Казань, Ростов, Екатеринбург, Воронеж и другие склады, где есть потребность.
3. Создаёт отдельную цель бронирования на каждый город/склад.
4. Запускает фоновых воркеров по целям одновременно, но с общим rate limit на Ozon-кабинет.
5. Для каждой цели создаёт или переиспользует черновик поставки.
6. Проверяет доступные таймслоты по складам цели.
7. Если подходящий слот найден, сразу пытается создать/подтвердить поставку.
8. Если Ozon возвращает `429`, задача не падает: воркер ждёт `Retry-After` или делает exponential backoff с jitter.
9. Если по складу нет потребности или нет подходящего слота, цель остаётся в поиске или пропускается по правилам задачи.
10. Пользователь видит статусы: `ожидает`, `ищет`, `слот найден`, `забронировано`, `лимит Ozon`, `ошибка`, `остановлено`.

## Важное ограничение

Автоматизация должна работать только через официальный Ozon Seller API. Нельзя делать скрытый браузер, автоклики по личному кабинету, обход капчи или агрессивный опрос без лимитов. Это риск блокировки кабинета продавца и самого сервиса.

Фраза "пробовать на все склады одновременно" в реализации означает:

- по каждому складу/городу есть независимая цель;
- цели проверяются параллельно через очередь;
- количество одновременных запросов ограничено настройкой `concurrency_limit`;
- между попытками есть пауза и случайный jitter;
- при `429` весь Ozon-аккаунт временно замедляется.

## API Ozon, который нужен для реального бронирования

Актуальные методы Ozon нужно подтвердить перед финальной интеграцией в коде, потому что FBO supply API активно меняется. По состоянию на май 2026 года для нового контура поставок ориентируемся на:

- создание черновиков: `/v1/draft/direct/create`, `/v1/draft/crossdock/create`, `/v1/draft/multi-cluster/create`;
- информация по черновику: `/v2/draft/create/info`;
- таймслоты черновика: `/v2/draft/timeslot/info`;
- создание поставки из черновика: `/v2/draft/supply/create`;
- статус создания поставки: `/v2/draft/supply/create/status`;
- информация по заявке: `/v3/supply-order/get`;
- интервалы существующей заявки: `/v1/supply-order/timeslot/get`;
- обновление интервала существующей заявки: `/v1/supply-order/timeslot/update`;
- статус обновления интервала: `/v1/supply-order/timeslot/status`.

Старые методы `/v1/draft/create`, `/v1/draft/create/info`, `/v1/draft/timeslot/info`, `/v1/draft/supply/create`, `/v1/draft/supply/create/status` объявлены устаревающими с отключением 16 марта 2026 года, поэтому текущий MVP создания черновиков надо перевести на новый v2/v3-контур перед продакшеном.

Источники для проверки перед разработкой:

- https://docs.ozon.ru/api/seller/
- https://t.me/s/OzonSellerAPI?before=642
- https://pypi.org/project/ozonapi-async/

## Алгоритм поиска слота

```text
Пользователь загрузил Excel
  -> сервис распознал SKU и количество
  -> сервис рассчитал распределение по городам
  -> пользователь нажал "Запустить поиск слотов"
  -> создаётся slot_booking_job
  -> создаются slot_booking_targets по городам/складам
  -> воркеры параллельно обрабатывают targets
       -> создать/обновить черновик
       -> запросить доступные таймслоты
       -> выбрать лучший слот по правилам
       -> создать поставку или обновить интервал
       -> сохранить результат
  -> задача заканчивается, когда:
       все цели забронированы,
       истёк max_wait_until,
       пользователь нажал "Остановить",
       Ozon вернул нерешаемую ошибку по цели.
```

## Правила выбора слота

По умолчанию выбираем слот так:

1. Сначала склады с максимальной потребностью по продажам и остаткам.
2. Потом крупные склады для новых товаров: Москва, Санкт-Петербург, Казань, Ростов, Екатеринбург, Воронеж.
3. Чем раньше дата, тем выше приоритет, если склад важный.
4. Если слот слишком маленький или склад не принимает нужный объём, слот пропускается.
5. Если товар не нужен в городе, цель по этому городу не создаётся.

Настройки задачи:

- максимальное время поиска: например 2 часа, 12 часов, 24 часа;
- интервал попыток: например 30-120 секунд;
- лимит одновременных целей: например 3-5;
- список городов/складов;
- минимальное количество SKU/штук для бронирования;
- режим: только уведомить или автоматически бронировать;
- стоп после первого успешного слота или продолжать расширять географию.

## UI

В рабочем кабинете нужен отдельный раздел `Слоты`:

- кнопка `Запустить поиск слотов` после расчёта поставки;
- список городов/складов с количеством SKU и штук;
- переключатель `Автоматически бронировать найденный слот`;
- поле `Искать до`;
- поле `Интервал проверки`;
- поле `Одновременных складов`;
- журнал попыток;
- кнопка `Остановить`;
- уведомление при успешном бронировании.

## Фоновая архитектура

Для локального MVP можно держать задачу в памяти процесса, но для настоящего веб-сервиса нужен такой контур:

- PostgreSQL хранит задачи, цели, позиции и попытки.
- Redis хранит очередь и rate limit по `ozon_account_id`.
- Worker забирает `slot_booking_targets` в статусе `waiting/searching`.
- Каждый запрос к Ozon идёт через общий `OzonRateLimiter`.
- Воркеры ставят heartbeat, чтобы зависшие задачи можно было восстановить.
- Все ответы Ozon с ошибками сохраняются в `slot_booking_attempts`.
- Пользователь видит прогресс через polling endpoint или WebSocket/SSE.

## Минимальные backend endpoints

```text
POST   /api/slot-booking/jobs
GET    /api/slot-booking/jobs/{job_id}
POST   /api/slot-booking/jobs/{job_id}/stop
GET    /api/slot-booking/jobs/{job_id}/attempts
```

Создание задачи должно требовать явного подтверждения пользователя. Автоматическое бронирование нельзя запускать тихо после простой загрузки Excel.

## Роли Ozon API

Для чтения товаров и аналитики достаточно read-only ролей, но для создания черновиков, поставок и выбора слотов понадобится роль с правами на FBO-поставки:

- `Supply order`;
- или `Admin`, если продавец согласен дать полный доступ.

Для публичного сервиса лучше просить минимальные права: товары/аналитика/склады для расчёта и `Supply order` только для пользователей, которые включают автоматическое бронирование.
